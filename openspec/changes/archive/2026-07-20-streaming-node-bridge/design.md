## Context

v2.0.0 delivered `stream_read` / `stream_write` in `src/stream.rs` (per-entry zip

+ quick-xml SAX, no `Workbook` model) and a napi surface in `src/stream_handle.rs`.
That surface aggregates the whole workbook at the FFI boundary:

```text
read(buffer)  -> stream_read(&[u8]) -> Vec<StreamSheet> -> Vec<JsStreamSheet> -> Promise<Array>
write(sheets) -> Vec<StreamSheet> -> stream_write -> Buffer -> Promise<Buffer>
```

Rust streams internally but the napi `read`/`write` collect everything, so JS
heap holds the entire workbook. The `streaming-xlsx` spec already requires
"readable byte stream" / "writable byte stream" I/O and incremental row yields;
v2.0.0 deferred the constant-memory Node bridge (design.md D4 / Open Questions,
GitHub #25.1).

Constraints discovered during exploration:
+ `zip::ZipArchive` requires `Seek` (reads the central directory at EOF) →
  **streaming READ from a non-seekable `Readable` is not free** today.
+ `zip::ZipWriter` writes sequentially to a `Write` sink (data descriptors) →
  **streaming WRITE to a `Writable`/file is feasible**.
+ `napi = 3` (features `async`, `napi5`): has `ThreadsafeFunction`, async fns,
  and `spawn_blocking` for off-event-loop CPU parsing.
+ `index.js` / `native.js` are **auto-generated by NAPI-RS** → stream adapters
  must live in a hand-written module.
+ Issue #25.3 note: "single-row Node streaming of arbitrarily huge sheets
  remains a later concern" → **row-level yields are explicitly deferred**;
  sheet-level is the intended target.

## Goals / Non-Goals

**Goals:**
+ Remove the `Vec<StreamSheet>` / full-`Buffer` aggregation at the FFI boundary.
+ Expose `read` as an `AsyncIterable<StreamSheet>` (one sheet in memory at a
  time) and as a Node `Readable`.
+ Expose `write` as accepting an `AsyncIterable<StreamSheet>` (constant-memory
  input) and streaming to a `Writable` / file / `Buffer`.
+ Pull-based backpressure (consumer paces the producer).
+ Non-breaking: existing `read`/`write` Promise overloads retained.

**Non-Goals:**
+ Row-level streaming of arbitrarily huge single sheets (deferred, #25.3).
+ Streaming READ directly from a non-seekable `Readable` byte source (blocked by
  `ZipArchive`'s `Seek` requirement; later spike — Strategy C / streaming zip
  reader).
+ Per-cell styles on the streaming path (stay on in-memory `xlsx`, v2.0.0).
+ Shared-formula member resolution (#25.2, separate change).

## Decisions

**D1 — Pull-based `AsyncIterable` primitive (over push/TSFN).**
Native exposes `StreamReader` / `StreamWriter` classes with `next()` /
`write_sheet()`; JS consumes via `for await`. Backpressure is inherent: the Rust
producer is idle between `next()` calls, so the consumer paces it with no bounded
channel or `ThreadsafeFunction` acknowledgement machinery.
*Alternative considered:* push model via `ThreadsafeFunction` + bounded `mpsc`
into a Node `Readable` (Strategy B). More moving parts for the same result; only
preferable if a pure push `Readable` is required without an iterator wrapper.

**D2 — Sheet-level granularity (over row-level).**
`read` yields one `StreamSheet` at a time; only the current sheet's rows are in
memory. This satisfies "without materializing the entire workbook in memory"
(the workbook = all sheets) and matches v2.0.0's no-`Workbook`-model design.
Row-level yields (one `StreamRow` at a time with sheet-boundary markers) are the
already-deferred "huge sheets" concern and out of scope here.

**D3 — Minimal native class + thin JS adapter layer.**
Rust exposes only the pull primitive; the three named shapes (`AsyncIterable`,
`Readable`, `Writable`) are produced in a hand-written JS wrapper:
`class implements [Symbol.asyncIterator]`, `Readable.from(iter)`, and a `Writable`
that drains an `AsyncIterable`. This keeps the Rust surface small and puts the
"Node stream shape" where it belongs (JS), since `index.js`/`native.js` are
generated.

**D4 — Read source: `Buffer` (existing) + file path (new, seekable).**
Both are seekable, so `zip::ZipArchive` works and JS heap stays small (disk-backed
file, or a provided `Buffer`). True non-seekable `Readable` input is deferred
(see Non-Goals / Risks).

**D5 — Write sink: `Buffer` (existing) + `Writable` / file path (new).**
`ZipWriter` emits each sheet entry to a `Write` sink incrementally (data
descriptors), so output streams to a `Writable`/file without buffering the whole
workbook.

**D6 — Off-loop CPU parsing.**
`next()` / `write_sheet()` run the SAX work via `spawn_blocking` (or an napi async
task) so a huge sheet parse does not block the Node event loop.

## Risks / Trade-offs

+ **`ZipArchive` needs `Seek`** → streaming read from a raw `Readable` is blocked.
  *Mitigation:* seekable source (`Buffer`/file) for #25.1; non-seekable read is a
  separate spike (streaming zip reader reading sequential local headers).
+ **`ZipWriter` to a non-seekable `Write`** (data descriptors) must be verified in
  `zip` 7 for Stored + Deflated. *Mitigation:* spike before committing to
  `Writable` output; fall back to temp spool file if a sink proves unsupported.
+ **Per-row napi object churn** is acceptable at sheet-level; revisit only if
  row-level is later added.
+ **Mid-stream error propagation**: hostile-input caps (#25.3) must abort the
  iterator with a clean JS error, not leak partial state. *Mitigation:* propagate
  `Err` out of `next()` / `write_sheet()`; ensure the producer thread is joined
  and the channel drained/closed on error.
+ **Exact napi-rs async-iterator expression** (next()-class + JS symbol vs a
  generated async generator) is unconfirmed. *Mitigation:* quick spike (Task 2.3)
  before the bulk of the Rust work.
+ **API break risk**: returning `AsyncIterable` instead of `Promise<Array>` would
  break callers. *Mitigation:* keep the `Promise` overloads (additive), so no
  breaking change.

## Migration Plan

Additive only. Existing `read(buffer): Promise<StreamSheet[]>` and
`write(sheets): Promise<Buffer>` continue to work; new streaming overloads are
added to the `stream` namespace. Regenerate `index.d.ts`; update README / ROADMAP;
close #25.1. No rollback needed beyond a normal release (no data/schema change).

## Open Questions

1. Keep the `Buffer`/`Promise` `read`/`write` overloads alongside the streaming
   forms (recommended, non-breaking), or replace them?
2. Cleanest napi-rs expression of the pull primitive: a `#[napi]` class with
   `next()` + a JS `[Symbol.asyncIterator]`, vs. a generated async generator?
3. Should `write` stream directly to a `Writable`, or accept an `AsyncIterable`
   and still return a `Buffer` (deferring `Writable` output to a follow-up)?
