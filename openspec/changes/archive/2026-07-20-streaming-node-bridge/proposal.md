## Why

v2.0.0 shipped a streaming XLSX reader/writer whose Rust core parses/serializes
incrementally (per-entry zip + SAX, no in-memory `Workbook` model), but the napi
surface defeats that gain at the FFI boundary: `read(buffer)` returns
`Promise<StreamSheet[]>` and `write(sheets)` takes `StreamSheet[]` then returns
`Promise<Buffer>` — the entire workbook is collected into JS memory before the
Promise resolves. The `streaming-xlsx` spec already mandates "readable byte
stream" / "writable byte stream" I/O and "yield worksheet rows incrementally";
# 25.1 is the deferred follow-up that closes the gap with a true constant-memory
Node `Readable` / `Writable` / `AsyncIterable` bridge.

## What Changes

- **Read bridge**: `wb.stream.xlsx.read(input)` returns an `AsyncIterable<StreamSheet>`
  (one sheet materialized at a time) instead of `Promise<StreamSheet[]>`. Also
  obtainable as a Node `Readable` via `Readable.from(iter)`.
- **Write bridge**: `wb.stream.xlsx.write(source)` accepts an
  `AsyncIterable<StreamSheet>` (constant-memory input) and streams output to a
  `Buffer` or a Node `Writable` / file path, instead of requiring the full
  `StreamSheet[]` up front.
- **Read input source**: keep `Buffer` input and **add a file-path input**
  (seekable, disk-backed) so the whole file is not forced into the JS heap.
- **JS adapter layer**: a new hand-written module adapts a minimal native
  pull-primitive into `AsyncIterable` / `Readable` / `Writable`. The generated
  `index.js` / `native.js` bindings must not be edited directly.
- **Scope**: values-only (numbers / strings / booleans / formulas). Per-cell
  styles stay on the in-memory `xlsx` path, consistent with v2.0.0.
- **Non-breaking**: existing `read(buffer): Promise<StreamSheet[]>` /
  `write(sheets): Promise<Buffer>` forms are retained as overloads; the streaming
  forms are additive to the `stream` namespace.

## Capabilities

### New Capabilities

- `streaming-node-bridge`: The Node-facing streaming contract over the existing
  `streaming-xlsx` Rust core — `read` yields an `AsyncIterable<StreamSheet>`
  (and a `Readable`), `write` accepts an `AsyncIterable<StreamSheet>` and streams
  to a `Writable`/Buffer, sheet-level granularity, seekable source (Buffer +
  file path). Fulfills the byte-stream requirements already stated in the
  `streaming-xlsx` spec that v2.0.0 deferred at the FFI surface.

## Impact

- `src/stream_handle.rs` (napi surface): replace aggregate `read`/`write` with a
  minimal pull-primitive (`StreamReader` / `StreamWriter` classes exposing
  `next()` / `write_sheet()`, parser driven off the main thread via
  `spawn_blocking`).
- `src/stream.rs`: refactor `stream_read` from "collect `Vec<StreamSheet>`" to
  produce one sheet at a time (sheet-level, parse state persisted across calls);
  add incremental `stream_write` entry emission to a `Write` sink
  (`ZipWriter` streaming with data descriptors).
- New hand-written JS module (e.g. `index.mjs` wrapper importing `./native.js`):
  `[Symbol.asyncIterator]`, `Readable.from(iter)`, and `Writable` adapters.
  `index.js` / `native.js` are napi-generated and untouched.
- `index.d.ts` regenerated by napi for the new signatures.
- Closes GitHub issue #25.1. No change to existing `read`/`write`/`stream`
  behavior for current callers.
