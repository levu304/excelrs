## Context

`excelrs` is a napi-rs addon. Its published entry (`index.js` / `index.d.ts`) is **auto-generated by `napi build`** — napi-rs has no template mechanism, so every run fully overwrites both files.

PR #17 ("v1.0.0 drop-in ExcelJS compatibility") added a **hand-written compat layer** patching two generated files:

- `index.js`: `Worksheet.prototype.getCell` / `Row.prototype.getCell` delegating to the Rust `getCellBy*` APIs.
- `index.d.ts`: `getCell` overload type declarations merged onto `Worksheet` / `Row`.

The streaming-bridge PR (#33) ran `napi build`, which regenerated `index.js` / `index.d.ts` from scratch and **wiped the glue**. CI's Typecheck job now fails with `TS2339: Property 'getCell' does not exist on type 'Worksheet'` across 6 test files — even though the runtime `getCellBy*` APIs still exist.

The fix must restore the glue in a way that **survives every future `napi build`**, not re-patch committed files by hand.

## Goals / Non-Goals

**Goals:**

- Restore `Worksheet.getCell` and `Row.getCell` overloads so ExcelJS drop-in callers keep working.
- Make the glue regenerate automatically on `napi build` (no manual re-patch step).
- Keep zero Rust changes and zero public-API changes.

**Non-Goals:**

- Not changing the Rust `getCellBy*` method names or signatures.
- Not adding new ExcelJS-compat surface beyond `getCell`.
- Not refactoring the napi build pipeline beyond the glue hook.

## Decisions

**D1 — Use napi-rs's `--pipe` post-build hook (Option A1).**

- `napi build --pipe "node scripts/apply-glue.cjs"` pipes each generated output file through the script. `apply-glue.cjs` appends the glue to `index.js` and `index.d.ts` in place.
- *Rationale*: `--pipe` is the **only** supported napi-rs mechanism for post-processing generated files (no `--template` / `--append` exists). It runs automatically as part of `build` / `build:release`, so the glue can never be forgotten.
- *Alternatives considered*:
  - **A2 — separate `build:glue` npm script** (manual append after build): simpler, but easy to forget; CI or local builds that skip it silently regress. Rejected.
  - **B — hand-written `index.js` wrapper importing `native.js`**: clean separation, but napi always regenerates the `index.js` entry (no `--no-index` flag), so the wrapper would be overwritten too. Rejected.
  - **C — `#[napi(js_name = "getCell")]`**: napi-derive supports 1:1 renames, but **cannot create overloads** (ExcelJS `getCell` accepts `string` *or* `number, number`; napi requires unique method names). Would also break internal Rust tests calling `get_cell_by_address`. Rejected.
  - **D — drop compat, update tests to `getCellByAddress`**: clean, but a breaking change to the documented ExcelJS drop-in promise. Rejected.

**D2 — Glue content matches the lost main-branch layer exactly.**

- `index.js`: append two prototype patches (delegating to `getCellByAddress`/`getCellByRc` for `Worksheet`, `getCellByColNum`/`getCellByColLetter` for `Row`).
- `index.d.ts`: append **`export interface Worksheet` / `export interface Row`** blocks declaring the `getCell` overloads. TypeScript **declaration merging** folds these into the generated `declare class`, so overloads typecheck without editing the class body (which would be wiped on regenerate anyway).

**D3 — Script is idempotent and file-scoped.**

- `apply-glue.cjs` inspects `process.argv` for the output file path, only acts on `index.js` / `index.d.ts`, and skips if the glue marker string is already present. Safe to run repeatedly.

## Risks / Trade-offs

- **[Risk] `--pipe` arg-passing shape unknown** → *Mitigation*: confirm during implementation by logging `process.argv` in a dry run; napi-rs docs show the form `napi build --pipe "npx prettier --write"`, i.e. command receives the file path. If it pipes via stdin instead, switch the script to stdin→stdout.
- **[Risk] Declaration merging requires the generated names to stay `Worksheet` / `Row`** → *Mitigation*: these are stable napi class names; `apply-glue.cjs` checks for their presence before appending.
- **[Trade-off] Glue lives in a script, not in the generated files** → slightly less obvious than inline patching, but this is the whole point: it must not live in regenerate-able files.
- **[Trade-off] `index.d.ts` keeps `@ts-nocheck`?** → napi emits `// @ts-nocheck` at the top; declaration merging still resolves at consumer typecheck time, so overloads are visible to downstream `tsc`. Verified by CI Typecheck passing post-fix.
