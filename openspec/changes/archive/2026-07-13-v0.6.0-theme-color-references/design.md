## Context

`excelrs` mirrors exceljs. Colors in the model are `Option<String>` ARGB/RGB
hex (style.rs, index.d.ts). The reader parses `xl/styles.xml` in three passes
and, at every `<color>` element, currently reads only the `rgb` attribute and
drops `theme`/`indexed` (styles.rs reader doc + the skip branches). The writer
emits `rgb="<ARGB>"` and needs no change.

## Goals / Non-Goals

- Resolve `theme="N"` and `indexed="N"` color refs to ARGB on read.
- Keep `color` a plain ARGB string ŌĆö zero public-API change.
- No write-side theme emission.

## Decisions

### D1. New `src/model/color.rs` holds `ThemeColorScheme`

A single resolver module, mirroring how `StyleTableRead` is a self-contained
model type. Keeps color math out of the parser. Fields:
`entries: [String; 12]` (dk1, lt1, dk2, lt2, accent1..6, hlink, folHlink ŌĆö each
6-char RGB; alpha forced to `FF` on resolve) and
`indexed: Option<[String; 56]>` (custom override from `<indexedColors>`).

### D2. Default scheme fallback (ECMA-376 ┬¦18.8.27)

When `xl/theme1.xml` is missing or unparseable, use the canonical default
`clrScheme` (dk1=000000, lt1=FFFFFF, dk2=1F497D, lt2=EEECE1, accent1=4F81BD,
accent2=C0504D, accent3=9BBB59, accent4=F79646, accent5=8064A2, accent6=4BACC6,
hlink=0000FF, folHlink=800080). This matches what Excel/ExcelJS assume for
files that omit a custom theme.

### D3. Tint algorithm (CT_Tint, ECMA-376 ┬¦18.8.3)

For each RGB channel `c` (0..255), `t = tint`:

- `t < 0` (darken): `c' = round(c * (1 + t))`
- `t >= 0` (lighten): `c' = round(c + (255 - c) * t)`
Clamp to [0,255]. Applied in `ThemeColorScheme::resolve_theme(index, tint)`.

### D4. No public API change

`color` stays `Option<String>`. A themed file read by excelrs yields the
resolved ARGB; writer emits it unchanged. This is the minimal, exceljs-compat
path and avoids a `Color` object migration (which would break every existing
test and caller).

### D5. Fixture strategy ŌĆö generate in-memory, parse custom schemes via inline XML

JS tests build themed `.xlsx` at runtime with ExcelJS (`Color { theme: N }` /
`{ indexed: N }` / `{ theme: N, tint: T }`), then load via excelrs ŌĆö no
committed binary needed for default-scheme cases. Custom-scheme parsing is
covered by Rust unit tests with inline `<a:clrScheme>` XML. One committed
fixture `fixtures/custom-theme.xlsx` (a hand-made workbook whose `theme1.xml`
uses non-default accents) is added only for the JS custom-scheme test; it can
be generated by a small throwaway script, not committed manually.

## Risks / Trade-offs

- **[Round-trip is color-resolved, not reference-preserving]** ŌåÆ acceptable;
  the ARGB model cannot express `theme="N"` without a `Color` object. Noted in
  Non-Goals.
- **[Existing 13 `parse_style_table` unit tests]** ŌåÆ signature change to add
  `&ThemeColorScheme` requires a mechanical update (pass `&default()`). Tracked
  as an impl task, not new tests.
- **[Tint precision]** ŌåÆ floating-point rounding may differ by ┬▒1 from Excel's
  exact value; JS tint assertions use tolerance, not equality.
- **[Custom indexed palette rare]** ŌåÆ only emitted by niche tools; default
  56-palette covers the common case.

## Migration Plan

- Semver minor (0.5.0 ŌåÆ 0.6.0). No caller-visible change; themed colors simply
  stop being `null`. Update spec ┬¦6.8 and ┬¦9.2.1 as follow-up doc task.
